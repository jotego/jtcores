# Purpose

Use this guide when a core needs to connect to the JTFRAME line-based frame
buffer family:

- `jtframe_lfbuf_ddr`
- `jtframe_lfbuf_cram`
- other wrappers built around `jtframe_lfbuf_line`

This skill is about the contract seen by the core. The external memory type is
handled by JTFRAME. From the core point of view, the interface is the same.

# Core-Side Model

The line frame buffer sits between the object or pixel generator and the final
video output.

The core does not write a full frame directly into external RAM. Instead it:

1. Receives line start timing from JTFRAME
2. Draws one line into the `ln_*` write port
3. Pulses `ln_done` when that line is complete
4. Lets JTFRAME copy that finished line into the target memory
5. Receives the previously stored frame back as `ln_pxl`

This makes the memory backend interchangeable while keeping the core-side
interface stable.

# Interface Signals

The usual core-facing signals are:

```verilog
input      [HW-1:0] ln_addr;
input      [DW-1:0] ln_data;
input               ln_we;
input               ln_done;
output              ln_hs;
output              ln_vs;
output              ln_lvbl;
output     [VW-1:0] ln_v;
output     [DW-1:0] ln_dout;
output     [DW-1:0] ln_pxl;
```

Interpret them as follows:

- `ln_hs`: line start strobe for the producer side. Use it to reset the
  line-local write state in the core.
- `ln_vs`: frame or vertical sync indication generated by the line-buffer
  timing path. This is only meaningful when full vertical timing is enabled.
- `ln_lvbl`: line-buffer vertical blank state. In minimal mode it may stay at
  its default value and should not be relied on.
- `ln_v`: line number that the core must render next.
- `ln_addr`: horizontal address written by the core inside the current line.
- `ln_data`: pixel value written by the core.
- `ln_we`: write enable for a valid pixel at `ln_addr`.
- `ln_done`: pulse when the whole requested line has been produced.
- `ln_dout`: readback value from the line-build buffer at the current
  `ln_addr`. Use it for read/modify/write effects such as shadows or blending.
- `ln_pxl`: pixel stream read back from the previously stored frame.

# What The Core Must Do

On every `ln_hs` pulse:

1. Start rendering the line identified by `ln_v`
2. Clear or reset the core's internal horizontal write pointer if it has one
3. Begin issuing `ln_we` writes for visible pixels only

While rendering the line:

1. Drive `ln_addr` with the destination X coordinate
2. Drive `ln_data` with the final pixel value for that X coordinate
3. Assert `ln_we` only for pixels that should overwrite the line buffer

When the line is finished:

1. Pulse `ln_done` for one cycle
2. Stop writing that line
3. Wait for the next `ln_hs`

The expected use is sparse writing. Transparent pixels should normally not be
written. `jtframe_lfbuf_line.v` explicitly notes that the core should not send
transparent pixels.

# Timing Rules

- `ln_hs` is the core-side line kickoff event.
- `ln_v` is valid for the line being requested by that `ln_hs`.
- `ln_done` should be a clean pulse after the last write for that line.
- JTFRAME captures the `ln_done` rising edge and starts the external-memory
  transfer from there.
- Do not keep `ln_done` asserted for multiple lines.
- `ln_dout` follows the core-side address path, not the scanout path.
- `ln_pxl` is synchronous to `clk` and updated on `pxl_cen`.

The line-buffer path double-buffers internally:

1. one line is being built from core writes
2. one completed line is being copied to frame storage
3. one line is being fetched back from frame storage
4. one fetched line is being scanned out as `ln_pxl`

Because of that, the displayed data corresponds to the previously stored frame,
not the line currently being written.
When the core needs random-access readback while building the line, use
`ln_dout` instead of `ln_pxl`.

# Coordinate And Width Expectations

- `HW` is the horizontal address width for one line
- `VW` is the vertical line index width
- `DW` must fit within 16 bits for the current JTFRAME implementation

`jtframe_lfbuf_line.v` stores core writes in a 16-bit internal RAM and stops in
simulation if `DW > 16`.

Match `ln_addr` to the scan-space used by `hdump`. In practice this means the
core should write pixels using the same horizontal indexing convention that the
line buffer later uses for scanout.

# Vertical Timing Behavior

JTFRAME derives the line-generation schedule from incoming video timing:

- `vs`
- `lvbl`
- `vrender`
- `hdump`
- `lhbl`

The wrapper converts those into core requests. The important core-facing rule is
that line generation starts from JTFRAME timing, not from a core-generated line
counter.

At the start of a frame request sequence, JTFRAME:

1. toggles the frame buffer bank
2. sets `ln_v` to the measured active-video start line
3. pulses `ln_hs` to request the first line

After each accepted `ln_done`, JTFRAME advances to the next requested line until
the measured active-video end line is reached.

# Full Vertical Mode

If `JTFRAME_LF_FULLV` is enabled, the wrapper also synthesizes vertical blank
and sync timing on the core side:

- `ln_vs`
- `ln_lvbl`
- blank-line handling through `fb_blank`

If `JTFRAME_LF_FULLV` is not enabled, only `ln_hs` and `ln_v` should be treated
as the reliable line-generation handshake for the core logic.

# Backend Independence

The core-side protocol should not change depending on whether the frame storage
uses:

- MiSTer DDR via `jtframe_lfbuf_ddr`
- Pocket Cell RAM via `jtframe_lfbuf_cram`
- another controller wrapped around `jtframe_lfbuf_line`

Backend-specific logic only affects the memory-side ports and the rate at which
JTFRAME can move finished lines around. The core still writes the same `ln_*`
signals and consumes the same `ln_pxl` output.

# Minimal Integration Pattern

Typical usage inside a core:

```verilog
always @(posedge clk) begin
    if (rst) begin
        x <= 0;
        ln_done <= 0;
    end else begin
        ln_done <= 0;
        if (ln_hs) begin
            x <= 0;
            render_busy <= 1;
            current_line <= ln_v;
        end else if (render_busy) begin
            if (pixel_valid) begin
                ln_addr <= pixel_x;
                ln_data <= pixel_data;
                ln_we   <= 1;
            end else begin
                ln_we   <= 0;
            end
            if (line_finished) begin
                ln_done <= 1;
                render_busy <= 0;
            end
        end else begin
            ln_we <= 0;
        end
    end
end
```

Adapt this to the core's renderer, but keep the handshake semantics unchanged.

# Common Mistakes

- Driving `ln_done` before all pixels for the line have been written
- Using `ln_pxl` for per-pixel read/modify/write instead of `ln_dout`
- Treating `ln_pxl` as the just-rendered line instead of the stored-frame output
- Writing transparent pixels instead of leaving them untouched
- Assuming `ln_vs` and `ln_lvbl` are meaningful when `JTFRAME_LF_FULLV` is off
- Using `DW > 16` without updating the shared line-buffer implementation
- Generating your own line schedule instead of following `ln_hs` and `ln_v`

# Practical Checklist

1. Instantiate the target-specific wrapper such as `jtframe_lfbuf_ddr` or
   `jtframe_lfbuf_cram`
2. Connect video timing inputs from the normal scan generator
3. Treat `ln_hs` plus `ln_v` as the render request
4. Write visible pixels through `ln_addr`, `ln_data`, `ln_we`
5. Pulse `ln_done` once per requested line
6. Use `ln_pxl` as the buffered video data returned by JTFRAME
7. Keep the core logic independent of the external RAM backend
