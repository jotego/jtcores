# Ground-up 005885 port — `jtddribble_k005885.v`

Status: **in progress** (started 2026-06-15). This is the planned REPLACEMENT for
the contra-derived `jtddribble_5885` + `jtddribble_5885_7121_{gfx,tilemap,obj}`
trio, which renders but has a render-pipeline pixel-cen PHASE bug (thin text,
every 2nd stroke pixel dropped — root-caused, see `fixes_journal.md`).

The new module is a faithful port of the **Iron Horse MiSTer model**
(`doc/k005885_REFERENCE.sv`, MIT, by Ace) — the authoritative ground-up 005885 —
adapted to JTFRAME. We instantiate **two** of them (E16=FG/gfx1, H16=BG/gfx2),
each producing `COL[4:0]`, combined by the existing `jtddribble_colmix` priority
mux + palette.

## Why a port of the Iron Horse model (not MAME-spec, not patch-contra)

- It is the real chip's RENDER TIMING — exactly what the contra-derived pipeline
  gets wrong. The thin-text bug is a timing/phase artefact; the reference has the
  correct pixel pipeline (tile_color_latch / tile_shift / tilemap_shift / 17-line
  sprite delay / cen_6m-latched COL).
- It produces `COL[4:0]` = one full chip → matches ddribble's 2-chip topology
  one-for-one (no architectural reshaping).
- konamiic.txt register doc embedded in the reference explicitly lists ddribble
  fg/bg scroll modes → it already covers our chip.

## The clock mapping (SOLVED — near drop-in)

| Reference (Iron Horse)        | ddribble JTFRAME            |
|-------------------------------|-----------------------------|
| `CK49` = 49.152 MHz           | `clk` = **48 MHz**          |
| `cen_6m = !div[2:0]` (clk/8)  | internal `div` clk/8 = 6 MHz (== framework `pxl_cen`) |
| `cen_3m = !div`    (clk/16)   | internal `div` clk/16 = 3 MHz |
| 8 sub-phases `div[2:0]`       | same `div[2:0]` 8 phases    |

`JTFRAME_PXLCLK=6` ⇒ `pxl_cen = clk/8`, `pxl2_cen = clk/4` (jtframe_pxlcen.v).
We **free-run our own `div[3:0]` at `clk`** exactly like the reference (CK49→clk
verbatim). It is a constant phase offset from the framework's `pxl_cen` — both are
exact clk/8 dividers, so the offset is harmless (our RGB is stable across a pixel;
the framework samples it at its own pxl_cen). The chip OWNS sync (HBLK/VBLK/HS/VS),
replacing `jtddribble_video.v`'s `jtframe_vtimer` at integration.

## The two real problems (NOT in the reference)

### 1. SDRAM latency — the reference assumes zero-latency ROM
The reference reads tile data (`RD<={RDU,RDL}` latched at h_cnt[1]) and sprite
data (`SD` combinational from `S`) as if the gfx ROM answers in ~1 clk. ddribble's
gfx is 256 KB (gfx1) + 512 KB (gfx2) in **SDRAM** (multi-cycle latency, jtframe
romrq 64-bit cache). So both ROM reads must become **registered, rom_ok-gated**
fetches:
- **Tiles**: fetched every 4 pixels (32 clk) — slow; prefetch + the 64-bit cache
  makes the adjacent-half read a hit. Low risk.
- **Sprites**: the reference fills the line buffer reading the sprite ROM at FULL
  clk (one word/clk in FSM state 6). SDRAM cannot sustain that. The sprite engine
  must fetch with handshaking (the whole line is available to fill the buffer, so
  a slower fill is fine — this is how the contra obj engine already works).

### 2. One time-shared gfx bus per chip (not the reference's separate R + S)
The real 005885 has ONE `R[15:0]` ROM bus; ddribble puts tiles in the lower half
and sprites in the upper half of each gfx region (`OBJSTART` offset). The reference
splits `R` (tiles) + `S` (sprites) into two parallel buses.

**Decision: keep ONE SDRAM read port per chip** (matches current mem.yaml/game.v —
clean swap, no mem.yaml change) and **time-multiplex** tile vs sprite fetches in the
module's fetch FSM, sprites addressed at `+OBJSTART`. The reference is the
BEHAVIOURAL spec (what addresses/sizes/flips/line-buffer the engine computes); the
FETCH MECHANISM is JTFRAME-style (shared SDRAM port, rom_cs/rom_addr/rom_data/rom_ok).

## RAM primitive mapping — external vs internal (DECIDED 2026-06-15)

The schematic shows **two EXTERNAL RAM chips** the 005885 drives; the CPU has NO
direct access to either (it reaches them only through the chip's bus). Per the user
decision they are modeled **per Jotego standards as mem.yaml dual-port BRAM,
instantiated in `jtddribble_video.v` — NOT inside `jtddribble_k005885.v`**. The chip
drives BOTH BRAM ports (port A = CPU-mediated, port B = render); no port is wired to
the raw CPU bus.

| RAM | schematic | where | how |
|-----|-----------|-------|-----|
| **VRAM** 8 KB (tile attr/code + sprite list) | 6264SL (AX/VO/NVOW/NROE) | **external** → mem.yaml BRAM in video.v | chip exposes `vram_cpu_addr/din/we/dout` (port A) + `vram_scn_addr/dout` (port B). DONE: ports wired in the chip; mem.yaml entry + video.v wiring land at P10. |
| **Sprite line-buffer** | 2× 4464 DRAM (FA/FD/NRAS/NCAS) | **external** → mem.yaml BRAM in video.v | added at P8 (sprite engine). |
| ZRAM0/1/2 + scratch (0x05-0xDF) | INTERNAL chip RAM (scroll latches) | **internal** | `jtframe_dual_ram` inside the chip — stays. |
| SPR_CACHE | NOT a schematic chip (model convenience) | **internal** | `jtframe_dual_ram` inside the chip (P8). |

Reference→JTFRAME primitive notes for the internal/line-buffer RAMs: `spram`→
`jtframe_dual_ram` (tied 2nd port); `dpram_dc`→`jtframe_dual_ram` (independent
`clk0`/`clk1`). The only non-mechanical adaptations are the reference's negedge /
`~CK49` usages (SPR_CACHE `clock_b=~CK49`, `lbuff_Din` negedge) — fold each into a
single-clock posedge pipeline stage.

## Colour path (ddribble specifics)

- ddribble has a **sprite LUT PROM** (I15, 256 B — mem.yaml `prom`, loaded via
  `prom_we`/`prog_addr`) but **no tilemap LUT PROM** (VCF grounded). So the
  reference's `VCF/VCB→VCD` tilemap LUT is **bypassed** (VCD = passthrough), and the
  `OCF/OCB→OCD` sprite LUT IS used (load I15 into BRAM).
- Each chip outputs `COL[4:0]` to the existing `jtddribble_colmix`, which does the
  LS157 PRI mux (`pri=(|g1col[3:0])&g2col[4]`) + xBGR_555 palette. Keep 007327 as-is.

## P7 finding — the Iron Horse tile-code composition is GAME-SPECIFIC

Do NOT port the reference's `R`/`ATR4`/`ATR5`/tile-code bit assignment verbatim —
it is **Iron-Horse-specific**:
- Reference: `code = {tile_ctrl[1:0], attr[7:6], code_byte}`, `ATR4/5 = attr[4:5]`
  used as external ROM bank bits, flip from `attr[5:4]`.
- ddribble (MAME `get_fg_tile_info`): `num = code_byte + attr[7:6]<<8 + attr[5]<<10
  + charbank[1]<<11`, `flipyx = attr[5:4]` (attr[5] reused: code bit 10 AND flipY).
- The composition also **differs per chip**: fg/chip1 tiles = 128 KB = 4096 tiles
  (12-bit num, `R[15:0]` only, `RA16`=sprite-half select); bg/chip2 tiles = 256 KB
  = 8192 tiles (13-bit num, needs one extra bank bit). `get_bg_tile_info` is the
  authority for chip2.

**So P7 is a HYBRID**, not a transcription:
- **Fetch addressing** (tile-code → rom_addr, RA16/RA17, scan address): take
  ddribble's PROVEN version (MAME `get_fg/bg_tile_info`, already realised by the
  contra `jtddribble_5885_7121_tilemap.v` bank-select parameterisation — the data
  path is byte-correct, that was never the bug).
- **Pixel output pipeline** (tile_color_latch → tile_shift → tilemap_shift 8→4 mux
  → tilemap_dly → COL latched on cen_6m): take the Iron Horse reference. THIS is
  the part that fixes thin text (the contra path's line-buffer-read phase is what
  drops every 2nd stroke pixel; the reference's shift-register pipeline doesn't).

Next action for P7: fetch MAME `ddribble.cpp` `get_fg_tile_info` + `get_bg_tile_info`
(+ scroll handling) at the pinned commit into `doc/`, confirm the per-chip bank
bits, then write the hybrid tilemap.

## TODO — move the chip (and the whole board) to a 24 MHz clock

Right now `jtddribble_k005885` runs on the JTFRAME 48 MHz `clk` and derives its
pixel timing from `pxl_cen` (clk/8); the model plays the reference's CK49 role at
48 MHz. The goal of this rework is to re-base the chip — and therefore the entire
ddribble core — onto the **24 MHz domain** (`clk24` = 24.576 MHz, already provided
because `cfg/common.def` sets `JTFRAME_CLK24` + `JTFRAME_PLL=jtframe_pll6144`), so
the board is a **single 24 MHz clock domain**.

Why: the real chip's master is **CK18 = 18.432 MHz** (page-0 osc → CK18 net), and
24.576 MHz is the nearest JTFRAME PLL grid to it; running everything from one slow
clock avoids the clk24-vs-clk48 cross-domain hazards (e.g. the YM2203 FM bug we
fixed, where a clk24-domain cen was sampled on clk48 and double-counted — see
doc/fixes_journal.md / reference_cen_clock_domain). It should also ease hardware
timing/resources.

Do it with **all available info**, not a quick port:
- Re-derive the internal cen chain from the real **CK18 18.432 MHz** divide chain
  (the schematic /12 → 1.536 MHz E, the 6/3 MHz pixel rates) rather than the
  current "cen_6m = pxl_cen" shortcut — recompute against clk24, not clk48.
- Cross-check the Ace reference (k005885_REFERENCE.sv, written for CK49) and the
  005885_implementation.md timing notes when remapping each enable to the 24 MHz base.
- Re-introduce the **`clk24` port** to the chip (it was removed in the staged-dev
  cleanup because the clk48 model never used it — `4ba73b396`) and move `clk`/cens,
  the video pipeline (`jtddribble_video`), and the VLM/sound clocking onto it too.
- Re-verify the whole chain: lint, the POST `M`-glyph thin-text canary, fine-scroll,
  and sprite alignment via scene replay — the render must stay pixel-exact after the
  clock move.
