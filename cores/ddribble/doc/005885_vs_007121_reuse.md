# Konami 005885 (ddribble) vs 007121 (contra) — reuse analysis

**Purpose:** settle, once, the question "should ddribble reuse / parameterize an
existing tested 005885/007121 engine instead of the ground-up `jtddribble_k005885`?"
so it isn't re-litigated. **TL;DR — already answered: keep `k005885`. The two
chips share an *architecture* (line-buffer tilemap + sprite line buffer), which
ddribble already reuses, but they differ *structurally* in the register map, the
tile-code composition, and the sprite format — and those differences are NOT
parameterizable. The contra modules stay as reference, not dependencies.**

---

## 1. The chips

`k007121.cpp` (MAME) calls the 007121 *"an evolution of the 005885, with more
features."* So **005885 ≈ 007121 minus features**, and the two are
bus/state-machine identical. That makes the 007121 the right *reference*, but the
"minus features" are exactly where a shared module breaks (below).

- **005885** (Double Dribble GX690): two chips, FG (E16) + BG (H16). 5 registers,
  hardcoded tile-code, 11-bit sprite code, external 6264SL VRAM, external 007327
  palette, 5-bit `COL[4:0]`.
- **007121** (Contra etc.): `cores/contra/hdl/jtcontra_gfx.v` (+ `_tilemap.v`,
  `_obj.v`), the JTFRAME-native engine shared by **6 cores** (contra, labrun,
  comsc, flane, castle, mx5k). 8 registers + 32B zure (per-row/col scroll), per-bit
  tile-code overrides, 14-bit sprite code, internal RAM.

## 2. The three attempts + the decision

| | approach | outcome |
|---|---|---|
| **A** | fork `jtcontra_gfx`, adapt for 005885 | rendered for months, but a **pixel-pipeline phase bug** (thin text — every 2nd stroke pixel dropped — and ~10px sprite-vs-tilemap misalignment) |
| **B** | reuse `jtcontra_gfx` via files.yaml + a `MODE_5885` parameter | abandoned — too many *structural* (not parametric) gates needed |
| **C** ✅ | **port the 007121 _line-buffer tilemap_ into a 005885-native module; keep the 005885's own registers / colmix / sprite path** | **`jtddribble_k005885.v` — the champion** |

**Decision 2026-06-16** (`STATUS.md:39-99`): adopt the 007121 *line-buffer tilemap
render* (the part that fixes the phase bug — render is decoupled from display via a
double-buffered line RAM, so fine scroll is clean, there's no edge-priming, and it
auto-aligns with the sprite line buffer), but **KEEP** the 005885's 5-register
decode, 007327 colmix, and sprite engine. Result: POST screen renders **pixel-exact**
vs MAME (corr 1.000); the 2700 gameplay screen renders correctly.

## 3. Why parameterizing the 007121 (Option B) was rejected — structural, not parametric

1. **Pixel pipeline** — the original failure. The contra *direct-serializer* path
   has a phase/edge-priming bug for the 005885 case; no amount of `MODE_5885` gating
   fixes it. (The *line-buffer* render does — and that's what was adopted.)
2. **Registers** — 005885 has **5** (`0..4`: scroll_y, scroll_x, scroll_ctrl,
   tile_ctrl, irq+flip); 007121 has **8 + 32B zure**. Different decoder shape.
3. **Tile-code composition** — 005885 is **hardcoded**
   `code = code_byte + attr[7:6]<<8 + attr[5]<<10 + charbank[1]<<11`; 007121 uses
   per-bit `codeN_sel`/`extra_*` muxes. Not the same logic, gated per-mode.
4. **Sprite format** — 005885 = **11-bit** code, `byte0→code[7:0]`,
   `byte1[2:0]→code[10:8]`, **16×16** base tiles, `byte4[4:2]` size modes; 007121 =
   **14-bit** code, different byte walk, **8×8-quadrant** tiles. A different ROM walk.

## 4. Per-module reuse verdict (the obj/tilemap split)

The 007121 is `jtcontra_gfx.v` (wrapper) → `jtcontra_gfx_tilemap.v` +
`jtcontra_gfx_obj.v`. Neither sub-module can be **instantiated directly**:

### `jtcontra_gfx_tilemap.v` — *mechanism* reusable (already reused), *module* not
- **Good:** generic interface — reads tile data via an external scan port
  (`scan_addr → attr_scan/code_scan`) and writes an external line buffer. That's why
  its render-FSM + double-buffer *pattern* was adoptable into k005885 (P7).
- **Blocker:** the tile-code path is 007121-only and **not configurable**.
  `bank[1..4] = attr_scan[3 + codeN_sel]` (lines 109-112); `codeN_sel` is 2 bits →
  reaches only `attr[3..6]`. The 005885 needs `code[9] = attr[7]` — **out of the
  mux's reach**. Plus `strip_*`, `txt_en`/`layout`/`no_txt`, `extra_*`, and a 13-bit
  code that can't be cleanly tied off.

### `jtcontra_gfx_obj.v` — not reusable; only the buffer primitive is shared
- The scan/SDRAM interface is generic, but the **sprite-format decode is a different
  chip**: `code[13:0]` (14-bit), byte walk `byte4[7:6]→code[13:12]` / `byte0→code[9:2]`
  / `byte1[1:0]→code[11:10]`, **8×8-quadrant** sub-tiles, `size_attr`-driven
  `height_comb`. None of that matches the 005885 (§3.4) and none is parameterizable.
- The **only** shared piece is the output primitive `jtframe_obj_buffer` — which
  ddribble **already uses**.

## 5. The reuse boundary that IS in the champion (maximal + correct)

| concern | source | shared? |
|---|---|---|
| tilemap render | 007121 line-buffer architecture (ported into `k005885` P7) | ✅ pattern |
| sprite line buffer | `jtframe_obj_buffer` (same primitive contra uses) | ✅ module |
| colour mixer | 007327 path (`jtddribble_colmix`; contra drives the same 007327) | ✅ chip |
| register decode | 005885 5-register map | ❌ ddribble-specific (correct) |
| tile-code / sprite-format decode | 005885 hardcoded mapping | ❌ ddribble-specific (correct) |

## 6. Conclusion / what NOT to do

- **Keep `jtddribble_k005885.v`.** It is the correct synthesis. Finish **P8–P10**
  (consolidate + verify the sprite engine + colour mixer). *(Note: the task list
  marks P8 "pending" while `STATUS.md:44` says 2700 already renders sprites —
  reconcile the actual sprite state before more work.)*
- **Do NOT** re-attempt Option B (parameterize/instantiate `jtcontra_gfx*`). It is a
  documented dead end (the §3 structural differences) and would put the 6 cores on
  `jtcontra_gfx` at regression risk.
- The retired `jtddribble_5885*` files live in `doc/` (git-tracked renames); the
  `jtcontra_gfx*` files stay in `cores/contra/` — both are **reference only**, not
  build dependencies.
